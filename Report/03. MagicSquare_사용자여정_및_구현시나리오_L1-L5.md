# Magic Square — 사용자 여정 및 구현 시나리오 (Level 1~5)

본 문서는 4×4 Magic Square 학습 프로젝트를 위한 **Epic → Journey → Story → Technical(Gherkin) → 검증** 정리본이다.  
프로젝트 불변은 `Report/4x4_마방진_문제정의_및_배경_보고서.md` 및 `.cursorrules`와 일치해야 한다.

---

## Level 1 — Epic (비즈니스 목표)

### EPIC — Business Goal

**Epic:** Invariant(불변조건) 기반 사고 훈련 시스템 구축

4×4 Magic Square를 **학습 도메인**으로 삼아, 불변조건·계약·TDD·리팩터링을 한 흐름으로 연습할 수 있는 소프트웨어와 연습 과정을 갖춘다.

### 목적

4×4 Magic Square 문제를 활용해 다음을 **의도적으로 반복**한다.

1. **Invariant 중심 설계 사고**  
   값 집합(1~16, 각 1회), 검사 선(4행·4열·주대각선 2개, 총 10선), 매직 합(34) 등 도메인 불변을 코드·테스트·문서에서 동일하게 유지하고, 변경 시 한곳에서 추적 가능하게 한다.

2. **Dual-Track TDD 적용**  
   도메인 규칙(엔티티)과 유스케이스(컨트롤)·경계(입출력)를 구분해, **규격 테스트**와 **계약 테스트**를 병행한다.

3. **입력/출력 계약 명확화**  
   Boundary에서 허용하는 입력 형식·오류 표현, Control/Entity에 넘기는 데이터의 전제 조건을 명시하고, **계약 테스트**로 고정한다.

4. **설계 → 테스트 → 구현 → 리팩터링 흐름 체화**  
   ECB(Entity–Control–Boundary) 방향을 지키며, Red–Green–Refactor와 역할 분리를 습관화한다.

### 성공 기준

| 구분 | 기준 |
|------|------|
| 도메인 로직 테스트 | Domain(엔티티 중심) 코드 **라인 커버리지 95% 이상** |
| 입력 검증·계약 | 정의한 입출력·검증에 대한 **계약 테스트 100% 통과** |
| 하드코딩 | 도메인 의미 있는 숫자·문자열을 산재 배치하지 않고, **명명된 상수·설정·단일 출처**로 관리 |
| 매직 넘버 | 격자 크기, 값 범위, 매직 합, 선 개수 등은 **이름 있는 상수/타입**으로만 참조 |
| 추적성 | 각 주요 불변이 **어떤 테스트(또는 테스트 그룹)**에 매핑되는지 식별 가능(Invariant → Test traceability) |

### 범위 힌트

- **포함:** 4×4 격자 판별·(필요 시) 생성·표시, ECB 레이어, pytest 기반 테스트.
- **제외(별 Epic):** 다른 격자 크기, “10선” 외 추가 마법 선 정의 변경(문서·불변 변경 시 별도 정리).

## Level 2 — User Journey (사용자 여정)

### Persona

| 항목 | 내용 |
|------|------|
| 역할 | 소프트웨어 개발 **학습자** |
| 현재 초점 | **TDD** 훈련 중(RED → GREEN → REFACTOR, 테스트 먼저) |
| 배경 지식 | **Clean Architecture** 또는 본 프로젝트의 **ECB** 개념을 익히는 단계 |

**동기:** “기능이 돌아가게만 만든다”가 아니라, **불변조건·계약·도메인**을 기준으로 설계하고 테스트로 고정하는 방식을 체득한다.

### Journey 개요

4×4 Magic Square를 연습 도메인으로 두고, 구현보다 먼저 **문제 정의·계약·도메인 경계**를 세운 뒤, **Dual-Track TDD**로 UI와 로직을 병렬로 올리고, 마지막에 **회귀**로 품질을 잠근다.

### Step 1 — 문제 인식

**“마방진을 구현한다”가 아니라**, 다음을 먼저 문장으로 고정한다.

- **무엇이 참이어야 하는가(Invariant)**  
  예: 값은 1~16이 각각 한 번, 검사 선은 4행·4열·주대각선 2개(총 10선), 공통 합은 34.
- **무엇이 이 시스템의 책임인가**  
  판별만인지, 부분 채움·빈칸 탐색·풀이까지인지 역할을 명시한다.
- **무엇이 범위 밖인가**  
  다른 격자 크기, 다른 “마법” 선 정의 등은 별 문제로 둔다.

### Step 2 — 계약 정의

1. **입력 스키마 정의** — 격자 표현(4×4 정수), 허용 타입·범위, 빈 칸(0) 규칙 등.
2. **출력 스키마 정의** — 마방진 여부, 실패 시 정보, 완성 결과(예: 길이 6, 1-index 좌표) 등.
3. **예외 정책 정의** — 잘못된 길이, 범위, 중복, 빈칸 개수 등에 대한 오류 타입·메시지·Boundary 표현.

### Step 3 — Domain 분리

| 구성 요소 | 역할(요지) |
|-----------|------------|
| **BlankFinder** | 값 0인 빈칸 좌표 탐색(row-major 순서). |
| **MissingNumberFinder** | 1~16 중 미등장 수(누락 2개) 탐색. |
| **MagicSquareValidator** | 완전 격자에 대한 마방진 판정(10선·합 34·중복·범위). |
| **Solver** | 두 빈칸에 누락 수를 배치하는 시도 순서 및 결과 반환. |

**의존 방향:** 순수 규칙은 Entity, 흐름 조율은 Control, 입출력은 Boundary.

### Step 4 — Dual-Track 진행

1. **UI RED** — Boundary 관점 계약·표시·오류 테스트.
2. **Logic RED** — Entity·Control 관점 불변·판정·탐색 테스트.
3. **GREEN** — 각 트랙 최소 구현.
4. **REFACTOR** — 녹색 위에서만 구조 정리, 관측 동작 불변.

### Step 5 — 회귀 보호

- 엣지 케이스, 입력 오류 케이스, 조합 실패·통합 시나리오 테스트 보강.

### 여정 완료 시 학습자가 말할 수 있는 것

- “이 불변은 **이 테스트**가 지킨다.”
- “입력은 **이 스키마**, 실패는 **이 정책**이다.”
- “UI와 도메인 테스트를 **동시에** 깔고, 리팩터는 **녹색 위에서만** 했다.”

## Level 3 — User Stories

### Story 1 — 입력 검증

**As a** 학습자  
**I want to** 입력이 **정확히 4×4 격자 규칙**을 만족하는지 검증되길 원한다  
**So that** 잘못된 데이터가 Domain으로 넘어가지 않도록 한다  

**Acceptance Criteria**

| # | 기준 |
|---|------|
| AC1 | 4행×4열이 아니면 예외(또는 계약상 거부). |
| AC2 | 빈칸(0) 개수가 **정확히 2가 아니면** 예외. |
| AC3 | 0을 제외한 채워진 칸에 **중복**이 있으면 예외. |
| AC4 | 채워진 칸 값이 **1~16 범위 위반**이면 예외. |
| AC5 | 실패는 구체적 예외 타입 또는 검증 결과 코드로 구분 가능(광범위 삼키기 금지). |

### Story 2 — 빈칸 탐색

**As a** 학습자  
**I want to** 빈칸(값 0)의 좌표를 정확히 알고 싶다  
**So that** 두 빈칸에 대한 조합 시도가 가능하다  

**Acceptance Criteria:** row-major 순서 유지, **정확히 2개** 반환.

### Story 3 — 누락 숫자 탐색

**As a** 학습자  
**I want to** 1~16 중 격자에 없는 수를 알고 싶다  
**So that** 두 빈칸에 넣을 후보 값을 확정할 수 있다  

**Acceptance Criteria:** 1~16 중 누락 **2개**, **오름차순** 반환, 결정적 동작.

### Story 4 — 마방진 판정

**As a** 학습자  
**I want to** 완전히 채워진 4×4 격자가 마방진인지 판정받고 싶다  
**So that** 10선·매직 합 규칙이 Validator에서 일관되게 적용된다  

**Acceptance Criteria:** 모든 행 합 동일, 모든 열 합 동일, 두 주대각선 동일, 공통 합 **34**, 1~16 각 1회.

### Story 5 — 두 조합 시도

**As a** 학습자  
**I want to** 작은 수→첫 빈칸·큰 수→둘째 빈칸 시도 후, 실패 시 **역순 배치**를 수행하고 싶다  
**So that** 한쪽 순서가 실패해도 정답을 찾을 수 있다  

**Acceptance Criteria:** 전진 시도(small→first blank, large→second), 실패 시 reverse, 정답 시 **길이 6 배열**(예: `[r1,c1,v1,r2,c2,v2]`, **1-index**), 두 시도 모두 실패 시 명시적 실패.

### 의존 관계(권장 순서)

`Story 1` → `Story 2`, `Story 3` (병렬 가능) → `Story 4` → `Story 5`

## Level 4 — 구현 시나리오 (Technical, Gherkin)

> 구현 시: 시나리오 1은 **전진 배치로 34를 만족하는 Given 보드**가 필요하다. 아래 예시 격자는 시나리오 2(역순)와 동일 원천 데이터를 쓰고 있어, 자동 검증 시 데이터를 분리·검증할 것.  
> 시나리오 2의 When은 **두 빈칸 모두 채운 뒤 Validator로 판정**하는 해석이 자연스럽다.

```gherkin
# language: ko
Feature: 4x4 마방진 완성
  불변조건 기반 로직을 검증하기 위해
  TDD를 연습하는 개발자로서
  부분적으로 채워진 4x4 마방진을 완성하고 싶다

  Background:
    Given 4x4 행렬이 주어지고
    And 0은 빈칸을 의미하며
    And 정확히 2개의 셀이 0을 포함하고 있고
    And 숫자는 1부터 16 사이여야 하며
    And 0을 제외한 중복 숫자는 허용되지 않으며
    And 4x4의 마방진 상수는 34이다

  Scenario: 작은 수 → 큰 수 순서로 마방진이 완성되는 경우
    Given 다음과 같은 행렬이 주어졌을 때:
      | 16 |  2 |  3 | 13 |
      |  5 | 11 | 10 |  8 |
      |  9 |  7 |  0 | 12 |
      |  4 | 14 | 15 |  0 |
    When 시스템이 빈칸 좌표를 찾고
    And 누락된 두 숫자를 찾은 뒤
    And 작은 숫자를 첫 번째 빈칸에 배치하고
    And 큰 숫자를 두 번째 빈칸에 배치하면
    Then 모든 행의 합은 34여야 하고
    And 모든 열의 합은 34여야 하며
    And 두 대각선의 합도 34여야 하고
    And 결과는 길이 6의 배열로 반환되어야 하며
    And 반환되는 좌표는 1-index 기준이어야 한다

  Scenario: 역순 배치 시 마방진이 완성되는 경우
    Given 다음과 같은 행렬이 주어졌을 때:
      | 16 |  2 |  3 | 13 |
      |  5 | 11 | 10 |  8 |
      |  9 |  7 |  0 | 12 |
      |  4 | 14 | 15 |  0 |
    When 작은 숫자를 첫 번째 빈칸에 배치했을 때 마방진이 되지 않고
    And 큰 숫자를 첫 번째 빈칸에 배치했을 때 마방진이 되면
    Then 시스템은 역순 배치 결과를 반환해야 하며
    And 최종 행렬은 마방진 상수 34를 만족해야 한다

  Scenario: 빈칸 개수가 올바르지 않은 경우
    Given 행렬에 빈칸이 1개만 존재할 때
    When 유효성 검증을 수행하면
    Then 오류가 발생해야 한다

  Scenario: 중복 숫자가 존재하는 경우
    Given 0을 제외한 중복 숫자가 포함된 행렬일 때
    When 유효성 검증을 수행하면
    Then 오류가 발생해야 한다

  Scenario: 값의 범위를 벗어난 경우
    Given 행렬에 16을 초과하는 숫자가 포함되어 있을 때
    When 유효성 검증을 수행하면
    Then 오류가 발생해야 한다
```
