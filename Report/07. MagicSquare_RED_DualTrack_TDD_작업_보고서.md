## 0. 테스트를 직접 돌리기 전에

### 0.1 필요한 것

- **Python 3.10 이상** (`pyproject.toml`의 `requires-python = ">=3.10"`)
- **pytest** (프로젝트는 `[project.optional-dependencies] dev = ["pytest>=7.0"]` 로 선언)

### 0.2 저장소 루트에서 작업한다

아래 모든 명령은 **프로젝트 루트** `Magic-Square_20`( `pyproject.toml` 이 있는 폴더)에서 실행한다.

Windows PowerShell 예시:

```powershell
Set-Location "c:\DEV\Magic-Square_20"
```

### 0.3 가상환경(venv)에서 실행하기

전역 Python을 건드리지 않으려면 **프로젝트 루트에 venv를 두고**, 활성화한 뒤 그 안에서 `pip`·`pytest`를 쓰면 된다.

**1) 한 번만: 가상환경 만들기** (루트에서)

```powershell
Set-Location "c:\DEV\Magic-Square_20"
python -m venv .venv
```

**2) 매 세션(터미널을 열 때마다): 활성화**

- **PowerShell** (권한 오류가 나면 아래 “실행 정책” 참고)

```powershell
Set-Location "c:\DEV\Magic-Square_20"
.\.venv\Scripts\Activate.ps1
```

프롬프트 앞에 `(.venv)` 가 붙으면 활성화된 상태다. 이후 `python`·`pip`는 모두 이 환경을 가리킨다.

- **명령 프롬프트(cmd)**

```cmd
cd /d c:\DEV\Magic-Square_20
.venv\Scripts\activate.bat
```

**3) 비활성화** (다른 프로젝트로 옮길 때)

```powershell
deactivate
```

**실행 정책**: `Activate.ps1` 이 막히면 관리자 PowerShell에서 한 번만 다음을 실행하거나, 사용자 범위로 완화한다.

```powershell
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
```

**폴더 이름**: `.venv` 대신 `venv` 등 다른 이름을 써도 되며, 해당 경로의 `Scripts\Activate.ps1` 만 맞추면 된다. `.venv`는 보통 `.gitignore`에 넣어 커밋하지 않는다.

### 0.4 의존성 설치 (venv 활성화 후, 택 1)

가상환경이 **활성화된 상태**에서 설치한다 (`pip`가 `.venv` 아래를 가리키는지 확인).

**방법 A — 선택적 dev 그룹으로 설치 (권장)**

```powershell
pip install -e ".[dev]"
```

**방법 B — pytest만**

```powershell
pip install "pytest>=7.0"
```

설치 확인:

```powershell
python -m pytest --version
```

### 0.5 `pythonpath`에 대해

[`pyproject.toml`](../pyproject.toml)에 다음이 있어, 루트에서 `python -m pytest`를 실행하면 **패키지 `entity`는 루트의 `entity/` 폴더**에서 로드된다.

```toml
[tool.pytest.ini_options]
testpaths = ["tests"]
pythonpath = ["."]
```

다만 **RED 단계**에서는 `entity.magic_square_ops`·`boundary.magic_square` **구현 파일이 없으므로** Entity/Boundary 테스트는 import 단계에서 실패한다(의도된 RED).

## 1. 작업 개요

1. **Dual-Track UI + Logic TDD** 관점에서, **Boundary(관측 계약)** 와 **Entity(순수 규칙)** 를 각각 pytest로 옮겼다.
2. **구현 코드는 작성하지 않았다.** `boundary/magic_square.py` 와 `entity/magic_square_ops.py` 는 없다. `entity/__init__.py` 만 패키지 마커로 존재한다.
3. **입력·출력 계약**: 부분 격자는 4×4 정수, `0`은 빈칸 **정확히 2개**, 채워진 값은 1~16·중복 없음. `solution` 성공 시 반환은 **길이 6의 `int` 리스트**, 좌표는 **1-index**, 형식 `[r1, c1, n1, r2, c2, n2]`. 매직 합은 **34** (Logic 쪽 `is_magic_square` 검증).
4. Logic 트랙은 테스트 명세·Level 4 Gherkin Given과 맞는 **고정 격자**를 사용한다.
5. **현재 기대 상태(RED)**: 루트에서 전체 `tests` 실행 시 **19건 실패**, 소요는 보통 1초 미만.
6. **문서 측 RED 명세**: 동일 계약을 사람이 읽기 쉬운 형태로 [`docs/RED_TestCase_Spec_MagicSquare_4x4_DualTrack.md`](../docs/RED_TestCase_Spec_MagicSquare_4x4_DualTrack.md)에 정리했다. UI-RED-01~07·LOGIC-RED-01~04, BOUND/LOGIC 테스트 ID·명세 이미지 TC-MS-* 대응, **구현·GREEN·REFACTOR 없음** 전제를 명시한다.
7. **원격 저장소**: 변경분은 브랜치 **`red`**에 커밋·푸시되었다. 원격: [https://github.com/jung-eunnnn/Magic-Square_20.git](https://github.com/jung-eunnnn/Magic-Square_20.git) (`origin/red`). 기본 브랜치가 `main`인 경우 GitHub에서 **`red`**를 선택해야 최신 RED·문서 커밋이 보인다.

## 3. RED일 때 보이는 실패 패턴

### 3.1 Boundary 트랙

- 대부분의 테스트는 `_solution()` → `from boundary.magic_square import solution` 경로로 진입한다.
- **전형적 오류**: `ModuleNotFoundError: No module named 'boundary'` 또는 `No module named 'boundary.magic_square'`.

### 3.2 Entity 트랙

- `_import_ops()` → `import entity.magic_square_ops as ops`.
- **전형적 오류**: `ModuleNotFoundError: No module named 'entity.magic_square_ops'`.

스텁 파일을 두고 `raise NotImplementedError`만 넣어도, import는 성공하므로 **“import 단계 RED”가 아니라 assertion RED**로 바뀐다. 본 프로젝트 RED 단계 설계는 **모듈 자체 부재**에 맞춰져 있다.
