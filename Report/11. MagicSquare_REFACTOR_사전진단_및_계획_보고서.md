## 3. ECB(Entity-Control-Boundary) 분석

### 3.1 현재 책임 배치

- `domain.py` / `boundary.py` / `gui/main_window.py` 파일명은 현재 저장소에 없고, 실질 대응 파일은 아래와 같다.
  - Domain 대응: [`entity/magic_square_ops.py`](../entity/magic_square_ops.py)
  - Boundary 대응: [`boundary/magic_square.py`](../boundary/magic_square.py)
  - GUI 대응: [`magicsquare/screen/app.py`](../magicsquare/screen/app.py)

- 판단 결과:
  - `entity/magic_square_ops.py`: **Entity + Control 혼재**
  - `boundary/magic_square.py`: **Boundary 역할 과다(도메인/유스케이스 로직 포함)**
  - `magicsquare/screen/app.py`: 대체로 UI 책임 유지, 단 도메인 범위값 일부 하드코딩 존재

### 3.2 잘못된 위치 코드 및 이동 대상

- `boundary/magic_square.py`의 `_is_magic_square`, `_find_blank_coords`, `_find_not_exist_nums`
  - 이동 권장: `entity` 계층(도메인 규칙/도메인 연산)
- `boundary/magic_square.py`의 `solution`
  - 이동 권장: `control` 계층(유스케이스 흐름 제어)
- `magicsquare/screen/app.py`의 값 범위 하드코딩(`0..16`)
  - 이동 권장: 도메인 상수 모듈(공용 상수 참조 구조)

### 3.3 Entity/Control 혼재 분리 포인트 (`entity/magic_square_ops.py`)

- Entity로 유지:
  - `find_blank_coords`, `find_not_exist_nums`, `is_magic_square`
- Control로 분리:
  - `solution` (후보 배치 순서 시도, 성공 케이스 결정, 반환 조립)

## 6. 테스트 선행 필요 항목

리팩터링 전에 아래 함수의 관측 동작을 테스트로 먼저 고정해야 한다.

- `boundary/magic_square.py`
  - `validate`: 예외 타입/조건, 빈칸 2개 규칙
  - `solution`: 반환 포맷(`[r1,c1,n1,r2,c2,n2]`), 1-based 좌표, 후보 시도 순서
- `entity/magic_square_ops.py`
  - `is_magic_square`: 합 판정, 0 포함 시 False
  - `find_blank_coords`: 빈칸 좌표 순서(행 우선)
  - `find_not_exist_nums`: 누락 숫자 정렬 반환
- `magicsquare/screen/app.py`
  - `_on_solve`: 성공/실패 시 라벨/경고 다이얼로그 동작
