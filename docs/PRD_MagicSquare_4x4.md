## 1. 방법론 골격: Dual-Track UI + Logic TDD

### 1.1 핵심 원리

- **UI(경계) 트랙**은 **UX Contract**(사용자에게 관측 가능한 상태·문구·가능/불가)로 RED를 설계한다.
- **로직 트랙**(Entity·Control)은 **Logic Rule**(불변·FR·입출력 계약)으로 RED를 설계한다.
- **GREEN·REFACTOR**는 두 트랙 모두에서 동일하게 적용하되, 테스트는 **서로의 구현 세부를 알지 않는다**.

### 1.2 “UI”의 본 PRD 해석 (범위 정합)

| 구분 | 의미 | 본 PRD 기본선 |
|------|------|----------------|
| 좁은 UI | 웹·데스크톱 GUI | **필수 아님** (`Report/...`, `.cursorrules`와 동일하게 DB·Web 비포함) |
| 넓은 Boundary | 사용자·운영자가 관측하는 **모든 경계** | **포함**: CLI 출력, 종료 코드, 로그/메시지 문자열, 향후 GUI가 붙을 때의 DOM·스냅샷 |

즉 **Dual-Track UI**는 “반드시 React”가 아니라, **관측 가능한 경계 계약(UX Contract)**과 **순수 규칙(Logic Rule)**을 테스트로 분리한다는 뜻으로 본다.

### 1.3 트랙별 책임·금지·검증 (ECB와 정렬)

**트랙 A — Boundary (UX Contract 층)**

| ID | 책임 | 금지 | 검증 |
|----|------|------|------|
| B-01 | 입력 수집, 결과·에러의 **관측 가능한 표면화** | 도메인 규칙을 Boundary에서 직접 구현하지 않음 | 계약 테스트: 허용 형식·거부 시 **문구/코드/상태**가 기대와 일치 |
| B-02 | Control 호출만으로 유스케이스 완성 | — | import: boundary → control → entity (`.cursorrules`) |

**트랙 B — Domain / Control (Logic Rule 층)**

| ID | 책임 | 검증 |
|----|------|------|
| D-01 | Entity: 격자 불변, 선별 합, 마방진 판정 등 순수 규칙 | 단위 테스트, 불변별 매핑 |
| D-02 | Control: 판별·탐색·두 조합 시도 등 절차 조율 | 통합·유스케이스 테스트 |
| D-03 | Entity는 boundary·control을 참조하지 않음 | 정적 의존성 또는 아키텍처 테스트(도입 시) |

### 1.4 두 트랙 테스트의 독립성 (필수 규칙)

- **Boundary 테스트**는 Entity/Control의 **내부 클래스 이름·알고리즘**에 의존하지 않는다. 공개 유스케이스 API 또는 시스템 경계에서의 관측만 사용한다.
- **Entity·Control 테스트**는 **DOM·터미널 색·프레임워크**에 의존하지 않는다. 입력 자료 구조와 규칙 결과만 사용한다.

## 2. MLOps와의 관계 (확장 훅)

### 2.1 기본선(현재 Epic·불변 범위)

4×4 마방진 **판별·검증·부분 채움**의 진실 공급원(Source of Truth)은 **결정적 규칙 엔진**(INV·FR·§10 계약)이다. **학습형 모델을 필수로 두지 않으며**, 본 PRD의 S-01~S-05 달성에 ML이 필요하지 않다.

### 2.2 ML 컴포넌트를 나중에 붙일 때(선택 확장)

다음 중 하나라도 도입하면 **동일 Dual-Track**을 유지하고, 아래 MLOps 최소 집합을 **별도 부록 또는 저장소 문서**로 구체화한다.

| 영역 | 요지 |
|------|------|
| 경계 | ML 출력(예: OCR 격자, “해 추천” 점수)은 **항상 규칙 엔진 앞 또는 병렬 검증** 대상. UX Contract는 “모델이 말한 것”이 아니라 **사용자에게 보이는 최종 허용/거부 상태**를 기술한다. |
| 데이터·아티팩트 | 학습·평가용 데이터셋 버전, 모델 버전, 재현 가능한 시드·해시를 식별 가능하게 둔다. |
| 품질 게이트 | CI에서 **규칙 기반 골든 테스트**와 **모델 회귀 지표**(정확도 등)를 분리해 게이트로 건다. |
| 운영 | 입력 분포 변화·오류율 상승 시 **모델만 재학습**하지 않고, PRD·불변·골든 케이스와의 **불일치(drift)**를 먼저 조사한다. |

### 2.3 “드리프트” 용어

- **도메인/코드 드리프트**: 구현·테스트가 PRD 불변·FR에서 벗어난 상태(추적 표 §13로 역추적).
- **데이터/모델 드리프트**: ML 도입 후에만 본격 적용; 규칙 엔진은 여전히 최종 판정의 기준선으로 둔다.

## 4. 용어

- **Dual-Track**: UX Contract 기반 **Boundary** 트랙과 Logic Rule 기반 **Entity·Control** 트랙을 분리해 테스트·계약을 병행하는 방식 (`Report/MagicSquare_사용자여정_및_구현시나리오_L1-L5.md` Level 1·2와 본 문서 §1).
- **UX Contract**: 경계에서 관측 가능한 언어 — 예: 특정 오류 문구가 **보인다/안 보인다**, 제출이 **가능하다/불가능하다**, 필드가 **활성/비활성**(GUI 시), 초기 상태에 오류가 **포함되지 않는다** 등.
- **Logic Rule**: 도메인 언어 — **허용/거부**, **유지/중단**, **반환/차단**, **계산**(합·누락 수) 등. **INV(§8)·FR(§9)·데이터·오류 계약(§10)**으로 표현한다.
- **ECB**: Entity–Control–Boundary (`.cursorrules` `architecture`).
- **검사 선(10선)**: 4행 + 4열 + 주대각선 2개 (`Report/4x4_마방진_문제정의_및_배경_보고서.md` §6.3, `.cursorrules` `project.invariants`).
- **C2C(Concept→Code) 추적**: 개념 → 규칙 → 유스케이스 → 계약 → 테스트 → 컴포넌트 연결(§13).
- **§6 vs §13**: **§6**은 동일 시나리오를 **사람이 읽기 쉬운 관측 언어(UX)·규칙 언어(Logic)**로 풀어 쓴 **시나리오 매핑 표**다. **§13**은 같은 내용을 **개발·회귀용으로** Concept·Rule·컴포넌트·테스트 유형까지 연결한 **추적 매트릭스**다. 중복이 아니라 읽는 목적이 다르다.

## 6. 시나리오 매핑: UX Contract ↔ Logic Rule

요구사항을 테스트로 내릴 때는 **판단(Decision)이 있는 서술**(허용/거부, 보임/안 보임 등)만 변환 대상으로 삼는다. 모호한 To-Do는 시나리오를 재작성한 뒤 매핑한다.

**§6.2와 §9의 관계**: 각 **MAP-ID** 행은 §9에 나열된 FR의 Acceptance Criteria를, 경계에서 **무엇이 관측되는지**(UX)와 도메인에서 **무슨 판단이 나는지**(Logic)로 한 번 더 풀어 쓴 것이다. FR만 단위 테스트로 담보하고 Boundary가 없는 구성이면, 해당 행의 UX 열은 “동일 조건을 공개 API 출력으로 재현할 때의 관측”으로 해석한다.

### 6.1 동사 언어 (참조)

| UX Contract (경계) | Logic Rule (도메인·유스케이스) |
|----------------------|--------------------------------|
| 보인다 / 안 보인다 | 허용한다 / 거부한다 |
| 가능하다 / 불가능하다 | 유지한다 / 중단한다 |
| 활성화 / 비활성화 (GUI) | 반환한다 / 차단한다 |
| 포함한다 / 포함하지 않는다 | 계산한다 (저장은 본 Epic 범위 외면 명시적으로 제외) |

### 6.2 Magic Square 3단 매핑 (Story·FR 정합)

아래 **UX Contract** 열은 “웹”이 아니어도 동일하게 **관측 가능한 경계**로 해석한다(§10.4 메시지 키·결과 코드·종료 코드 등). **MAP-ID**는 추적·이슈·테스트 이름 접두에 재사용할 수 있는 안정 식별자다.

| MAP-ID | 시나리오(요약) | FR | UX Contract (관측) | Logic Rule |
|--------|----------------|-----|---------------------|------------|
| MAP-01 | 격자 크기가 4×4가 아님 | FR-1.1 | 검증 실패 메시지 또는 결과 코드가 **보인다** | 구조 검증이 **거부** |
| MAP-02 | 빈칸(`0`) 개수가 정확히 2가 아님 | FR-1.2 | 빈칸 개수 오류 메시지·코드가 **보인다** | 빈칸 개수 규칙이 **거부** |
| MAP-03 | 채워진 칸에 값 중복 | FR-1.3 | 중복 오류 메시지·코드가 **보인다** | 중복 검사가 **거부** |
| MAP-04 | 채워진 값이 1~16 범위 밖 | FR-1.4 | 범위 오류 메시지·코드가 **보인다** | 범위 검증이 **거부** |
| MAP-05 | FR-1 통과 후 빈칸 좌표 나열 | FR-2 | (구조화 출력 시) 두 좌표가 row-major 순서로 **보인다** / API면 동일 순서로 **반환**됨이 검증 가능 | 빈칸 탐색이 좌표를 **반환** |
| MAP-06 | FR-1 통과 후 누락 두 수 나열 | FR-3 | 두 수가 오름차순으로 **보인다** / API면 동일 순서로 **반환** | 누락 집합이 **계산**되어 정렬 **반환** |
| MAP-07 | 완전 격자, 10선이 마방진 조건 불만족 | FR-4 | “판정: 비마방진” 등 결과가 **보인다** | 행·열·주대각 일치 검증이 **거부**(유효하지 않음) |
| MAP-08 | 완전 격자, 10선 모두 합 34 등 조건 만족 | FR-4, INV | “판정: 마방진” 등 결과가 **보인다** | Validator가 규칙에 따라 **허용**(유효) |
| MAP-09 | FR-1 통과 부분 격자에서 두 순서 시도 후 성공 | FR-5 | “완료·해” 류 피드백 또는 길이 6 결과가 **보인다** | FR-5 절차 후 **반환** |
| MAP-10 | 두 시도 모두 실패 | FR-5.4 | 실패가 **숨겨지지 않는다**(§10.2); 코드·예외로 **보인다** | FR-5.4 **명시적 실패** |
| MAP-11 | (GUI 확장 시) 비정수 입력 | FR-1.* / 파싱 | 해당 필드가 오류 상태로 **보인다** / 제출 **불가** | 파싱·검증이 입력을 **차단** 또는 **거부** |

### 6.3 P-05 검증 체크리스트 (요약)

## 10. 계약 (입력·출력·오류)

### 10.1 데이터 표현 (고정)

| 빈칸 | 정수 `0` |
| 완전 격자 | `0` 없음, 1~16 각 정확히 1회 |

### 10.2 오류 정책

| 조건 | 기대 동작 |
|------|-----------|
| FR-1 위반 | 거부; 구체적 유형 또는 코드로 구분 가능 |
| FR-5.4 | 실패를 숨기지 않음 |

### 10.3 Gherkin과의 관계

`Report/MagicSquare_사용자여정_및_구현시나리오_L1-L5.md` Level 4에 수록된 Feature/Scenario는 **행동 계약**의 참조 본문이다. Level 5 체크리스트에서 “Story 4 전용 시나리오 보강 권장”으로 표시된 항목은 **미완 보강 항목**이지 신규 기능 요구가 아니다.

구현 시 디렉터리는 `.cursorrules` `file_structure`를 따른다.

### 10.4 Boundary 관측 규약 (UX Contract 동결)

Boundary 계층이 존재하거나(CLI·GUI·공개 진입점) 나중에 추가될 때, **테스트가 단언할 관측량**은 코드베이스 안에서 **단일 출처**를 갖는다. 아래는 최소 집합이며, 구체 상수·열거형 이름은 구현과 함께 `src/` 또는 README에 고정해도 되나, **의미 변경 시 본 PRD §6·§10을 함께 갱신**한다.

| 관측량 | 규칙 |
|--------|------|
| **메시지 키** | 사람이 읽는 문자열만 두지 않고, **안정 키**(예: `validation.grid_shape`, `validation.empty_cells`, `validation.duplicate`, `validation.range`, `result.not_magic`, `result.magic`, `solver.success`, `solver.failed`)를 정의한다. Boundary 테스트는 가능하면 **키 또는 키+기본 로케일 한 문구** 중 하나로 단언한다. |
| **결과 코드** | 성공·FR-1 위반 종류·FR-5.4 명시적 실패 등을 구분할 수 있는 **열거형 또는 정수 코드**를 둔다. §10.2의 조건과 **N:1 또는 1:1 매핑**이 추적 가능해야 한다. |
| **프로세스 종료 코드**(CLI) | 성공 `0`, 검증 거부·솔버 실패는 `0`이 아닌 값으로 구분한다. 구체 정수는 구현 시 문서화하고, PRD에서는 “구분 가능”만 요구한다. |
| **공개 진입점 시그니처** | 애플리케이션 서비스·`main` 등 **공개** 함수의 입·출력·예외 타입은 NFR-04(docstring)와 일치하며, Logic Track은 동일 시나리오를 **Entity/Control 직접 호출**로도 재현 가능해야 한다(§1.4). |

Boundary 테스트는 **내부 클래스명·비공개 헬퍼·DOM 구현 세부(CSS 클래스 난수 등)**에 의존하지 않는다. GUI 도입 시에는 **역할·라벨·접근 가능 이름** 등 안정 셀렉터 또는 계약된 `data-testid`만 사용한다.

## 13. Concept → Rule → Use Case → Contract → Test → Component 추적성

§6.2 **MAP-ID**는 아래 표의 Concept·Rule과 같은 행을 가리킬 때 테스트 이름·커밋 메시지에 붙이면 **PRD ↔ 코드** 역추적이 쉬워진다. §6는 **읽기용 시나리오 매핑**, 본 절은 **구현·감사용 추적**이다.

### 13.1 추적 매트릭스 (핵심)

| Concept (개념) | Rule (불변·정책 ID) | Use Case (사용자 가치) | Contract (고정 조항) | Test (검증 유형) | Component (책임 주체, ECB) |
|----------------|---------------------|-------------------------|------------------------|-------------------|------------------------------|
