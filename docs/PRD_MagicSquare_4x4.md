## 2. 제품 개요

### 2.1 한 줄 설명

4×4 격자에 1~16을 각각 한 번씩 배치할 때, 고정된 10개 선에서의 합이 모두 동일한지 판별하고, 문서화된 역할 범위 안에서 생성·표시·(문서화된) 부분 채움 시나리오를 다루는 **학습용** 소프트웨어의 요구사항을 고정한다.

### 2.2 제품 목적 (우선순위: 알고리즘 난이도보다 TDD)

| ID | 목적 | 검증 가능 표현 |
|----|------|----------------|
| P-01 | TDD(Red–Green–Refactor) 절차를 반복해 습관화한다 | `.cursorrules` `tdd_rules`의 각 phase 규칙을 만족하는지 점검 가능 |
| P-02 | Invariant 기반 설계·검증 사고를 체득한다 | §6 불변 각 항목이 §10 추적 표에 대응하는 테스트 식별자를 갖는다 (`Report/...L1-L5.md` Epic 성공 기준) |
| P-03 | 입력·출력 계약을 명시하고 테스트로 고정한다 | §7 계약 항목마다 실패/성공을 판정하는 자동화 테스트 존재 여부로 검증 |
| P-04 | UI·DB·Web 없이 순수 로직 중심 구현이 가능하다 | 아키텍처 의존성: `entity`에 프레임워크·I/O 없음 (`.cursorrules`); 저장소에 DB·HTTP 요구사항 없음 |

### 2.3 범위 내 / 범위 외

| 구분 | 내용 | 출처 |
|------|------|------|
| 포함 | 4×4, 값 1~16 각 1회, 10선, 매직 합 34, 판별·(필요 시) 생성·표시, ECB, pytest | `.cursorrules` `project.summary`, `Report/...L1-L5.md` 범위 힌트 |

## 4. Dual-Track 요구사항 구조

요구사항은 **두 트랙**으로 읽는다. 구현 시 디렉터리는 `.cursorrules` `file_structure`를 따른다.

### 4.1 트랙 A — Boundary (경계)

| ID | 책임 | 금지 | 검증 |
|----|------|------|------|
| B-01 | 입력 수집, 결과·에러 메시지 표면화 | 도메인 규칙을 Boundary에서 직접 구현하지 않음 | 계약 테스트: 허용 입력 형식·거부 시 관측 가능한 오류 표현 |
| B-02 | Control 호출만으로 유스케이스 완성 | — | import 방향: boundary → control → entity (`.cursorrules`) |

### 4.2 트랙 B — Domain / Control (도메인·유스케이스)

| ID | 책임 | 검증 |
|----|------|------|
| D-01 | Entity: 격자 불변, 선별 합, 마방진 판정 등 순수 규칙 | 단위 테스트, 불변별 매핑 (Epic) |
| D-02 | Control: 판별·생성·탐색 등 절차 조율; 격자 불변 검증을 Boundary에 두지 않음 | 통합·유스케이스 테스트 |
| D-03 | Entity는 boundary·control을 참조하지 않음 | 정적 의존성 검사 또는 아키텍처 테스트(도입 시) |

## 6. 기능 요구사항 (기존 Story 범위 유지)

각 요구사항은 **Acceptance Criteria** 또는 동등한 Gherkin Then으로 검증 가능해야 한다.

### 6.1 FR-1 입력 검증 (Story 1)

| ID | 요구사항 | 검증 기준 (측정 가능) |
|----|----------|------------------------|
| FR-1.1 | 격자는 4행×4열 구조만 허용 | 행·열 수 ≠ 4인 입력 시 계약상 거부 또는 명시적 예외 |
| FR-1.2 | 값 `0`은 빈칸; `0`의 개수는 정확히 2 | `0` 개수 ≠ 2이면 거부 |
| FR-1.3 | `0`을 제외한 채워진 값에 중복 없음 | 중복 존재 시 거부 |
| FR-1.4 | 채워진 값은 1 이상 16 이하 정수 | 범위 밖 값 시 거부 |
| FR-1.5 | 실패는 구체적 예외 타입 또는 구분 가능한 검증 결과 코드로 표현 가능 | 광범위 `except` 후 무분별 삼키기 금지 (`.cursorrules` `forbidden`) |

### 6.2 FR-2 빈칸 탐색 (Story 2)

| ID | 요구사항 | 검증 기준 |
|----|----------|-----------|
| FR-2.1 | 빈칸은 값이 0인 셀 | Story 정의와 일치 |
| FR-2.2 | 반환 좌표는 row-major 순서 (행 오름차순, 동일 행이면 열 오름차순) | 동일 입력에 대해 좌표 순서 고정 |
| FR-2.3 | FR-1을 통과한 입력에 대해 빈칸 좌표는 정확히 2개 | 개수 = 2 |

### 6.3 FR-3 누락 숫자 탐색 (Story 3)

| ID | 요구사항 | 검증 기준 |
|----|----------|-----------|
| FR-3.1 | 1~16 중 격자에 없는 수는 정확히 2개 | 집합 차집합 결과 크기 = 2 |
| FR-3.2 | 두 누락 수는 오름차순으로 반환 | 리스트 정렬 검증 |
| FR-3.3 | 동일 입력 → 동일 출력 | 결정성 |

### 6.4 FR-4 마방진 판정 — 완전 격자 (Story 4)

| ID | 요구사항 | 검증 기준 |
|----|----------|-----------|
| FR-4.1 | 모든 행의 네 수 합이 서로 동일 | 4개 행 합 동일 |
| FR-4.2 | 모든 열의 네 수 합이 서로 동일 | 4개 열 합 동일 |
| FR-4.3 | 주대각선 2개의 합이 위 공통 합과 동일 | INV-02 선 집합과 일치 |
| FR-4.4 | 공통 합은 34이며 값은 1~16 각 1회 | INV-01, INV-03 |

### 6.5 FR-5 두 조합 시도 (Story 5)

| ID | 요구사항 | 검증 기준 |
|----|----------|-----------|
| FR-5.1 | 첫 시도: 누락 수 중 작은 수 → row-major 첫 빈칸, 큰 수 → 두 번째 빈칸 | 배치 후 FR-4로 판정 |
| FR-5.2 | 첫 시도가 마방진이 아니면 두 번째 시도: 역순(큰 수→첫 빈칸, 작은 수→둘째 빈칸) | `Report/...L1-L5.md` Level 4 주석: 두 빈칸 모두 채운 뒤 Validator로 판정하는 해석으로 스텝 정의 고정 |
| FR-5.3 | 성공 시 결과는 길이 6의 일차원 배열 `[r1,c1,v1,r2,c2,v2]`; 좌표는 1-index | 길이=6, 좌표 체계 검증 |
| FR-5.4 | 두 시도 모두 실패 시 명시적 실패 | 예외 또는 Result 타입 등 계약에 서술된 방식으로 관측 가능 |

### 6.6 프로젝트 요약와의 정합

`.cursorrules` `project.summary`의 “판별·생성·표시”는 본 PRD의 FR-4(판별) 및 Boundary 표시 책임과 모순되지 않는다. “생성”은 별도 유스케이스로 문서화된 뒤 구현하며, **부분 채움 완성(Story 5)**은 `Report/...L1-L5.md`에만 구체화되어 있으므로 본 PRD는 해당 문서 범위를 벗어나지 않는다.
