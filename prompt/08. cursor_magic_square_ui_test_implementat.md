## 한 GREEN 단계로 한 일

RED에서 요구하던 **`boundary.magic_square.solution`** 계약을 맞추기 위해 **`boundary/magic_square.py`** 한 모듈에 다음을 넣었습니다.

1. **입력 검증 (도메인으로 넘기기 전)**  
   - `None` → `TypeError`  
   - 셀이 정수가 아님 → `TypeError` (`type(x) is int`로 `bool` 혼입 방지)  
   - 4×4 아님·행 길이 불일치 → `ValueError`  
   - 빈칸(`0`) 개수가 정확히 2가 아님 → `ValueError`  
   - 0이 아닌 값이 1~16 밖 → `ValueError`  
   - 0 제외 중복 → `ValueError`

2. **풀이**  
   - 빈칸 좌표는 행 우선, 누락 수는 오름차순 `[a, b]`  
   - FR-5와 entity RED 스펙에 맞춰 **먼저 `(a,b)` 배치**, 매직이 아니면 **`(b,a)`** 시도  
   - 완성 격자에 대해 행·열·두 대각선 합이 34인지 검사

패키지로 인식되도록 **`boundary/__init__.py`**만 최소 내용으로 추가했습니다.

핵심 진입점은 아래와 같습니다.

```python
def solution(grid: list[list[int]]) -> list[int]:
    """Return ``[r1,c1,n1,r2,c2,n2]`` with 1-based row/col; two missing numbers placed."""
```

이후 Track B GREEN에서 `entity.magic_square_ops`를 만들면, boundary는 검증만 두고 `ops.solution`에 위임하도록 줄이면 됩니다.
