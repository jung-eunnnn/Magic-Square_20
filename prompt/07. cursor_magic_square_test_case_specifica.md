당신은 Dual-Track UI + Logic TDD 전문가입니다.

우리는 이미 Magic Square(4x4) 시나리오를 작성했습니다.
지금 단계는 구현이 아니라,
시나리오를 “실패하는 테스트(RED)”로 변환하는 것입니다.

중요
- 구현 코드를 작성하지 마십시오.
- GREEN 단계로 가지 마십시오.
- REFACTOR 단계로 가지 마십시오.
- 오직 RED 테스트만 작성하십시오.
- 모든 테스트는 현재 구현이 없기 때문에 실패해야 합니다.

------------------------------------------------------------
프로젝트 조건
------------------------------------------------------------

입력:
- 4x4 int\\\[]\\\[]
- 0은 빈칸
- 정확히 2개의 0
- 값 범위 1\\\~16
- 중복 금지 (0 제외)

출력:
- int\\\[6]
- 좌표는 1-index
- \\\[r1, c1, n1, r2, c2, n2]

마방진 상수:
- 4x4 → 34

------------------------------------------------------------
Dual-Track RED 단계 구조
------------------------------------------------------------

UI RED        Logic RED
   ↓              ↓
(둘 다 실패 상태)

------------------------------------------------------------
TRACK A — UI / Boundary RED
------------------------------------------------------------

다음을 실패 테스트로 변환하십시오:

1. 4x4가 아닌 입력은 예외 발생
2. 빈칸이 2개가 아니면 예외 발생
3. 값 범위 위반 시 예외 발생
4. 중복 숫자 존재 시 예외 발생
5. 반환 배열 길이는 6이어야 한다
6. 반환 좌표는 1-index여야 한다

각 테스트에 대해:
- 테스트 이름
- Given (입력)
- When (호출)
- Then (기대)
- 이 테스트가 보호하는 계약 설명

------------------------------------------------------------
TRACK B — Logic RED
------------------------------------------------------------

다음을 실패 테스트로 작성하십시오:

1. find\\\_blank\\\_coords()
   - row-major 순서로 정확히 2개 반환

2. find\\\_not\\\_exist\\\_nums()
   - 누락 숫자 2개 반환
   - 오름차순

3. is\\\_magic\\\_square()
   - 모든 행 합 동일
   - 모든 열 합 동일
   - 두 대각선 합 동일
   - 34와 일치

4. solution()
   - small → first blank 시도
   - 실패 시 reverse 시도
   - 6-length 반환
   - 1-index 좌표

각 테스트는:
- 실패 상태임을 명확히 설명
- 어떤 Invariant를 검증하는지 명시

------------------------------------------------------------
출력 형식
------------------------------------------------------------

# UI RED Tests
- Test ID
- Scenario
- Invariant 보호 내용

# Logic RED Tests
- Test ID
- Scenario
- Invariant 보호 내용

------------------------------------------------------------

구현 금지
GREEN 단계 금지
REFACTOR 금지
오직 RED 테스트 설계만 수행하십시오


참고해서 테스트 케이스 생성해줘 오직 RED 테스트 설계만 수행하십시오.
테스트케이스는 docs폴더에 만들어줘

**포함 내용 요약**

1. **# UI RED Tests**  
   - UI-RED-01 ~ 06: 요청하신 6가지(비 4×4, 빈칸 개수, 범위, 중복, 길이 6, 1-index)에 각각 **테스트 이름 예시, Given/When/Then, 계약 설명, Scenario, Invariant**를 표 형식이 아닌 절 단위로 정리했습니다.  
   - **UI-RED-07**: 기존 `test_magic_square_ui_red.py`와 맞춘 타입 계약(`None`, 비정수 → `TypeError`)을 보강 항목으로 넣었습니다.  
   - 명세 이미지의 **TC-MS-*** 와 **BOUND-MS-UI-*** 매핑 표를 두었습니다.

2. **# Logic RED Tests**  
   - `find_blank_coords`, `find_not_exist_nums`, `is_magic_square`, `solution`에 대해 동일한 Gherkin·Invariant·**RED가 되는 조건(import 실패·단언 불일치)**을 적었습니다.  
   - `solution`은 **작은 수 → row-major 첫 빈칸**, 실패 시 **역순 시도**, 길이 6·1-index·채운 뒤 `is_magic_square` 참까지 **LOGIC-MS-L-004**와 같은 Given/Then을 명시했습니다.  
   - `find_blank_coords`는 프로젝트 테스트가 **내부 0-index 튜플**을 쓰는 점을 문서에 한 줄로 밝히고, 팀이 1-index로 통일할 경우 단언만 바꾸면 된다고 적어 두었습니다.

3. **문서 하단**  
   - `tests/boundary/test_magic_square_ui_red.py`, `tests/entity/test_magic_square_logic_red.py`와의 **추적 링크**와 RED 단계 전제를 짧게 정리했습니다.

구현·GREEN·REFACTOR는 하지 않았고, **docs에 RED 테스트 설계만** 반영했습니다.
