슬라이드에 나온 **Dual-Track(UI는 UX 계약으로 RED, 로직은 규칙으로 RED, GREEN·REFACTOR는 함께)** 방향은 설계·테스트 관점에서 타당하고, 지금 PRD의 **Boundary vs Entity·Control** 구분과도 잘 맞습니다. 다만 **“with MLOps”**를 같은 문장에 붙이면, 지금 저장소의 Magic Square 4×4 범위와는 **겹치는 부분과 억지로 끼워 넣기 쉬운 부분**이 갈립니다.

## Dual-Track UI + Logic TDD — 의견

- **UI 트랙**: “렌더 직후 오류 문구 없음”, “빈칸 오류 시 특정 메시지 보임”처럼 [보인다/안 보인다, 가능/불가] 식 **UX Contract**를 테스트 제목·Given-When-Then에 박아 두면, 슬라이드 말대로 RED가 명확해집니다.
- **로직 트랙**: 마방진 불변·FR·계약을 **허용/거부, 반환/차단** 언어로 쓰면, 지금 PRD의 §5~§7과 그대로 연결됩니다.
- **완전 분리**: “UI 테스트는 도메인 클래스 이름을 모른다 / 로직 테스트는 DOM을 모른다”는 원칙은 유지보수·병렬 개발에 유리합니다. C2C(개념→규칙→유스케이스→계약→테스트) 표와도 잘 짝입니다.

**주의할 점**: 현재 `PRD_MagicSquare_4x4.md`는 **UI·DB·Web 없이 순수 로직**을 전제로 하고 있습니다. Dual-Track을 “진짜로” 쓰려면 PRD에 **Boundary(UI 또는 CLI)** 층과 그에 대한 계약·테스트 디렉터리를 범위에 넣는 **범위 확장**이 필요합니다. 슬라이드의 React 예시와 지금 문서는 그 지점에서 한 번 정리해야 합니다.

## MLOps를 같이 쓰고 싶다는 부분 — 의견

4×4 마방진의 **판별·검증·규칙 기반 완성**은 전형적으로 **결정적 알고리즘**이 맞고, “모델 학습 → 배포 → 모니터링”인 MLOps의 핵심 가치(데이터 드리프트, 재학습 파이프라인 등)와 **자연스럽게 맞물리기 어렵습니다.**  
그래서 선택지는 대략 세 가지입니다.

1. **MLOps는 빼고** “Dual-Track UI + Logic TDD + C2C 추적”만 명시 — 이 프로젝트 목적(TDD·불변·계약)과 가장 잘 맞습니다.  
2. **교육용으로만 MLOps 맛보기**: 예를 들어 “격자 이미지 OCR로 숫자 읽기” 같은 **별도 ML 경계**를 두고, 그 출력만 기존 로직 계약에 넣기 — 그때는 **ML 경계 = 불확실성 있는 Boundary**, 로직 트랙은 여전히 규칙으로만 검증하는 식으로 역할을 나누는 게 낫습니다.  
3. **“드리프트”를 도메인 비유로만**: 슬라이드 비교표의 “Drift detection”을 **코드/요구사항이 PRD에서 얼마나 벗어났는지** 같은 메타 관점으로 쓰는 것 — 가능하지만, 그건 통상 말하는 MLOps와는 다른 이야기입니다.

정리하면, **Dual-Track UI + Logic TDD + PRD↔테스트 추적**은 강하게 추천할 수 있고, **MLOps**는 이 도메인만으로는 동기가 약하니, “왜 ML인가?”가 한 문장으로 답이 될 때만 범위에 넣는 편이 낫습니다.

원하시면 다음 중 어디로 가실지 정해 주시면 그에 맞춰 PRD에 넣을 절 목차만 짧게 제안할 수 있습니다: **(A) UI 없이 Logic + Boundary(CLI)만**, **(B) 웹 UI까지 Dual-Track**, **(C) ML 보조 입력까지 포함한 3층 Boundary**.

## 2. 강점

1. **범위 충돌 처리**: §1.2에서 Dual-Track의 “UI”를 Boundary 전체로 정의해, **pytest만 있는 저장소**와 **나중 GUI**를 같은 PRD 틀로 설명할 수 있습니다.

2. **테스트로 내리기 위한 필터**: §6 첫 문단(판단·Decision 있는 서술만 테스트화)은 슬라이드의 “동사로 끝나지 않으면 테스트 금지”와 같은 긴장을 PRD에 넣은 부분입니다.

3. **MLOps의 역할 한정**: §2.1에서 ML 비필수·규칙 SoT를 박아 두고, §2.2는 “도입 시 할 일 목록” 수준이라 **현재 Epic을 부풀리지 않습니다.**

4. **추적성**: §13 매트릭스가 §8~10의 ID와 맞물려 있어, **전통 PRD 대비 “코드/테스트와의 연결”이 강합니다.**

5. **리스크**: §14에 방법론 오해(GUI만 UI)와 MLOps 이중 진실을 넣어, 나중에 읽는 사람이 빠지기 쉬운 구멍을 문서 차원에서 막습니다.

---

## 3. 약점·모호한 점

1. **§6.2 매핑의 커버리지**: 행은 대표 시나리오 위주이고, **FR-2·FR-3·FR-4 세부 행**이 표에 직접 매핑되어 있지는 않습니다. “§6 표의 각 셀”(P-05)과 비교하면 **표는 요약 수준**이라, 완전한 1:1 추적은 §9·§13에 더 의존하는 형태입니다.

2. **용어 한 줄(§4 Logic Rule)**: “FR·INV·§10 계약”으로 적혀 있는데, **FR 본문은 §9**입니다. 의미상 문제는 없으나, 엄밀히는 “§8~§10” 또는 “INV·FR(§9)·데이터/오류 계약(§10)”처럼 쓰면 내부 참조가 더 맞습니다.

3. **Boundary 테스트의 구체 표준**: §1.3은 “문구/코드/상태”까지 언급하지만, **메시지 카탈로그(키·로케일)**나 **공개 유스케이스 API의 시그니처**는 §10에 아직 없습니다. CLI만 있을 때 “UX Contract”를 무엇으로 동결할지는 구현 단계에서 추가 규약이 필요할 수 있습니다.

4. **§13과 §6의 역할 중복**: 둘 다 “개념↔검증”에 가깝습니다. §6은 **행동·관측 언어**, §13은 **컴포넌트·테스트 유형**으로 역할이 나뉘어 있긴 하나, 처음 읽는 사람은 **한 번은 “§6 = 사람이 읽는 시나리오 표, §13 = 개발자 추적 표”**라고 머릿속에 라벨을 붙이는 게 좋습니다.

## 변경 요약

1. **§4 용어**  
   - Logic Rule 정의를 **INV(§8)·FR(§9)·데이터·오류 계약(§10)**으로 정확히 표기.  
   - **§6 vs §13**: 시나리오용 매핑 vs 개발·추적용 매트릭스 역할 구분 문단 추가.

2. **§5 P-05**  
   - **MAP-ID** 단위로 Boundary 또는 Logic 테스트가 있는지 검증하도록 문구 수정.

3. **§6**  
   - §9와의 관계(각 MAP 행 = FR AC의 UX·Logic 풀이) 설명 추가.  
   - **§6.2**: `MAP-01`~`MAP-11` 열, **FR** 열, 시나리오를 FR-1.1~1.4, FR-2, FR-3, FR-4, FR-5 등으로 채움.  
   - **§6.3**: P-05용 체크리스트(MAP-01~10, MAP-11 조건부).

4. **§10.4 Boundary 관측 규약**  
   - 메시지 키 예시, 결과 코드, CLI 종료 코드, 공개 진입점 시그니처 규칙.  
   - Boundary 테스트가 **안 써야 할 것**(내부 클래스, 불안정 DOM 등) 명시.

5. **§12 S-02**  
   - MAP-ID ↔ 테스트 링크 테이블을 **권장**으로 한 줄 추가.

6. **§13**  
   - 도입부에 MAP-ID와 표의 연결·용도 설명.

7. **§15 변경 관리**  
   - §10.4까지 동시 개정하도록 수정.

8. **§16 문서 이력**  
   - 같은 날 보강분을 한 행으로 합침.

추가로 `MAP-11`의 `FR-1.*` 표기는 “파싱이 FR-1 이전에 있을 수 있음”을 넓게 쓴 것이므로, 구현에서 파싱을 별도 Story로 쪼개면 PRD의 FR 열만 조정하면 됩니다.
